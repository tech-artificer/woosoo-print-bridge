# 1. Mission Context

You are building an Android “relay device” app that connects to a Bluetooth thermal printer and prints orders coming from a Laravel backend in real-time.

This app runs on an Android device (tablet/phone) placed near the printer and acts as a 24/7 autonomous print station for a restaurant POS system.

**Printer details:**
- Brand/Model: GOOJPRT PB-58H
- Bluetooth PIN: 0000
- Interface: USB + Bluetooth
- Power: 9V / 2A
- Target paper size: 58mm

**Backend option:**
- WebSocket URL base: `ws://192.168.100.85:6001`
- Backend: Laravel 12 with Reverb (WebSockets)
- Relay listens on channel `admin.print`

**Flutter baseline ZIP:**
- Project: `woosoo_relay_device_clean.zip`
- Uses: `blue_thermal_printer`, `web_socket_channel`, `sembast` (queue persistence), Riverpod (state), polling fallback

# 2. Goal

Deliver a reliable, zero-touch Android app that:
- Receives print jobs via WebSocket (primary) and Polling (fallback)
- Deduplicates jobs using `print_event_id`
- Persists the print queue (survives crash/reboot)
- Prints receipts to PB-58H via Bluetooth (paired first, PIN 0000)
- Acknowledges print result to backend:
  - success: `/api/print-events/{id}/printed`
  - failure: `/api/print-events/{id}/failed`
- Retries prints automatically up to 3 times (exponential delay: 1s, 2s, 4s)
- Has a small UI:
  - Status (printer + queue counters)
  - Queue list (retry failed, cancel pending)
  - Settings (API/WS URL, printer pairing selection)
  - Logs viewer

**Non-goals:**
- No fancy template designer
- No complex multi-printer routing yet
- No iOS/Windows support needed right now (but code should remain portable)

# 3. Stack & Constraints

**Frontend Relay App:**
- Flutter (Dart)
- State: Riverpod
- Storage: Sembast
- Bluetooth: blue_thermal_printer
- Network: http, web_socket_channel
- System: permission_handler, wakelock_plus

**Backend (Existing):**
- Laravel 12
- Reverb WebSocket server
- Inertia + Vue + ShadCN + Tailwind (admin UI side, optional)

**Networking:**
- API Base URL: `http://192.168.100.85:8000`
- WS Base URL: `ws://192.168.100.85:6001`
- Channel: `admin.print`
- Event: `order.printed` (may arrive as `order.printed` or `.order.printed`)
- Payload must include: `print_event_id`, `order_id`, `device_id` (or fallback from config)

If backend does not yet have print-events endpoints, agent must implement them (Laravel task list included below).

# 6. Agent Tasks (Flutter)

## A. Stabilize Bluetooth Printing (PB-58H)
- Ensure Android permissions are correct for Android 12+: `BLUETOOTH_SCAN`, `BLUETOOTH_CONNECT`
- Location may be required for discovery on some devices
- Ensure device pairing flow is clear:
  - Pair printer in Android settings with PIN 0000
  - App only lists bonded devices and connects by MAC address
- Implement auto-reconnect:
  - If `printerAddress` is stored, attempt connect if disconnected

## B. Queue Processing Hardening
- Only process one pending job at a time (avoid overlapping prints)
- Ensure: `pending → printing → success/failed`
- Retry logic:
  - attempt 1 fail → pending (retry=1) delay 1s
  - attempt 2 fail → pending (retry=2) delay 2s
  - attempt 3 fail → failed (retry=3) and call failed endpoint

## C. Deduplication Rule
- Deduplicate strictly using `print_event_id`
- If payload is missing `print_event_id`, do NOT print (log and skip)

## D. WebSocket Handling
- Handle event name variants: `order.printed` and `.order.printed`
- Handle data possibly being a stringified JSON
- Subscribe channel `admin.print` using Pusher protocol message:
```json
{ "event": "pusher:subscribe", "data": { "channel": "admin.print" } }
```

## E. Polling Fallback
- Poll every 30s when WS is unstable or even in parallel
- Use `since` timestamp after first poll
- Dedup ensures safe parallel usage

## F. Settings UX
- Editable:
  - API Base URL
  - WS URL (must allow inserting real app key)
  - printerId (backend logical ID)
- Printer selector:
  - list bonded devices
  - tap to connect and save MAC/name

## G. Logging
- Log key lifecycle events:
  - startup/init
  - auth/register
  - ws connect/disconnect
  - poll tick results
  - enqueue job
  - print success/fail
  - ack success/fail

# 7. Agent Tasks (Laravel Backend Updates)

If backend is missing or inconsistent, implement these:

## A. Print Events Table + Model
- Create `print_events` table:
  - id (pk)
  - device_id (string or fk to devices)
  - order_id (fk or int)
  - session_id (nullable)
  - print_type (INITIAL/REFILL)
  - refill_number (nullable int)
  - payload (json)
  - status (pending/printed/failed)
  - attempt_count (default 0)
  - printed_at (nullable)
  - failed_at (nullable)
  - error (nullable text)
  - timestamps

## B. API Endpoints
- `GET /api/print-events/unprinted`
  - params: session_id, since, limit
  - returns unprinted/queued items
- `POST /api/print-events/{id}/printed`
  - idempotent
  - sets printed_at, status=printed
- `POST /api/print-events/{id}/failed`
  - sets failed_at, status=failed, error, attempt_count
- `POST /api/printer/heartbeat`
  - store last_seen_at, printer status, app_version, queue size, etc.

## C. WebSocket Broadcast
- Broadcast on channel `admin.print`
- event: `order.printed`
- include `print_event_id` in payload always
- ensure payload is consistent with Flutter expectations

## D. Device Lookup by IP
- `/api/device/lookup-by-ip` returns:
  - device_id
  - auth_token
  - printer_name
  - bluetooth_address (optional)

# 8. Definition of Done

A build is considered complete when:
- Android app can pair + connect PB-58H and print a test receipt
- WebSocket event received → job enqueued → printed → backend ack called
- App restart does not lose queued jobs
- Printer disconnect mid-run does not lose jobs; prints resume after reconnect
- Dedup works: same `print_event_id` cannot print twice
- Polling fallback works if WS is down
